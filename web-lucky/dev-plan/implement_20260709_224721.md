작성 일시: `2026-07-09 22:47:21 KST`

## 개발 목적

기존 `web-lucky` 토정 단일 앱을 **허브(홈)에서 모드를 고르는 통합 웹**으로 확장한다.

사용자가 첫 화면에서 기능을 고른 뒤, 아래 **5가지**를 각각 수행한다.

| # | 모드 ID | 이름 | 상태 |
|---|---|---|---|
| 1 | `iljin` | 오늘의 일진 대시보드 | **신규** |
| 3 | `chemi` | 우리 케미 점수 | **신규** |
| 4 | `naming` | 이름 후보 오행 체커 | **신규** |
| 5 | `solar-terms` | 절기 타임라인 | **신규** |
| 8 | `tojeong` | 토정비결 한 해 요약 | **기존** (리팩터로 모드 편입) |

- 계산은 모두 **`manseryeok-engine` 직결** (MCP HTTP 필수 아님).
- 공통: 오락·자기성찰 고지, 서버 저장 없음, 한국어 UI.
- 기존 토정 시연(1990-03-15 / 2026) 회귀를 깨지 않는다.

## 개발 범위

### A. 허브 · 라우팅 · 앱 셸

- **홈(`/` 또는 `view=home`)**: 5개 모드 카드 선택 UI (아이콘·한 줄 설명·CTA)
- **모드 진입**: URL 쿼리 권장 — `?mode=iljin|chemi|naming|solar-terms|tojeong`
  - 새로고침·공유 가능; history API로 뒤로가기 = 홈
- **공통 셸**: 상단 타이틀/「홈으로」, 푸터 고지 문구, 로딩·에러 패턴 공유
- 기존 `main.ts` 단일 폼을 **모드 라우터 + 모드별 모듈**로 분리

### B. 모드별 기능 (MVP)

#### B1. `iljin` — 오늘의 일진 대시보드

| 항목 | 내용 |
|---|---|
| 입력 | 조회 날짜 (기본: KST 오늘), 이전/다음 날 버튼 |
| 엔진 | `Calendar.getCalendarDay(y, m, d)` |
| 표시 | 양력·음력, 일진(간지), 오행, 12신살, 길흉, 택일 힌트, 당일 절기(있으면) |
| 비범위 | 월간 전체 달력 그리드 편집, 길흉 단정 코칭 |

#### B3. `chemi` — 우리 케미 점수

| 항목 | 내용 |
|---|---|
| 입력 | 사람1·사람2 생년월일, 양/음력·윤달, **성별** (엔진 궁합 입력), 출생시 선택(모름 허용) |
| 엔진 | `Compatibility.calculateCompatibility({ person1, person2 })` |
| 표시 | 총점·등급, 카테고리 점수(일간/일지/오행/구성 등 결과 필드), advice 목록 |
| 톤 | 낮은 등급도 “결함” 아닌 “리듬 차이”; 앱 프레임 고지 |
| 비범위 | 상대 목록 DB, 공유 이미지 고퀄 |

#### B4. `naming` — 이름 후보 오행 체커

| 항목 | 내용 |
|---|---|
| 입력 | 성(1자), 이름 후보 1~6개 (쉼표/줄바꿈 구분) |
| 엔진 | `Naming.analyzeNames(surname, givenNames)` |
| 표시 | 후보별 `totalScore`, 획수, 원형이정, 81수리 길흉, 발음/수리 오행, 정렬(점수 내림) |
| 비범위 | 한자 확장(`analyzeNamesExtended`) 필수화 — **선택 Phase**에서만; MVP는 한글 기본 분석 |

#### B5. `solar-terms` — 절기 타임라인

| 항목 | 내용 |
|---|---|
| 입력 | 연도 (기본: KST 올해), 이전/다음 해 |
| 엔진 | `listSolarTermsForYear(year)` (+ 필요 시 `getSolarTermsOnDate` / 현재 절기 판별 헬퍼) |
| 표시 | 24절기 세로/가로 타임라인, **다음 입절** 하이라이트·D-day, “지금 구간” 표시 |
| 비범위 | 절기 기반 운세 문장 생성(LLM) |

#### B8. `tojeong` — 기존 토정 (유지·편입)

| 항목 | 내용 |
|---|---|
| 입력·결과 | 기존 MVP 동일 (총운 + 12칸 + 공유 텍스트 + 음력 각주) |
| 작업 | 홈에서 진입 가능하게 분리; 도메인 `runTojeongYearly` 재사용 |

## 문서 기반 제약 사항

- 모든 날짜 “오늘/올해”는 **Asia/Seoul** (`kst.ts`) 우선.
- 토정·일진은 출생 시각 불필요; 케미는 엔진이 성별·(선택) 시각을 받음 — 모름 시 엔진 허용 값으로 명시 매핑.
- 양력/음력 변환이 필요한 모드는 변환 결과를 UI에 투명 표시(가능 시).
- 공유·클립보드에 불필요한 민감 생일 나열 금지(모드별: 토정은 기존 정책 유지; 케미 공유는 MVP에서 생략 또는 점수만).
- 엔진 원문 길흉 문구가 강해도 앱 셸 고지로 상쇄.
- `analyzeNames` 후보 **최대 6개** 엔진 제한 준수.

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 요구사항이 불명확하면 가능한 케이스를 나누고 확인 후 진행한다.
- 한 기능은 유지보수 가능한 최소 책임 단위로 나누고, 각 단위의 역할과 경계를 계획에 적는다.
- 계획 밖 옵션/확장/부가 기능을 만들기 위한 분리는 하지 않는다.
- 공통화는 계획 범위 안의 중복을 줄일 때만 수행하고 과도한 추상화는 피한다.
- 기존 프로젝트 API, 공식 SDK, 표준 라이브러리를 우선 검토하고 새 의존성은 범위상 필요할 때만 포함한다.
- 성공 정의: 홈에서 5모드 진입·완료 가능, 토정 회귀 통과, 모드별 단위 테스트(실엔진) 그린, `npm run build` 성공.
