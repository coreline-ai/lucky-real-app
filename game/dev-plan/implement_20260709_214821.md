## 개발 목적

브라우저 미니게임 `game/`(*일진 수호신 카드 배틀*)이 **실제 `manseryeok-mcp` 서버**를 호출해 플레이어 일주·오늘 일진을 받아 배틀에 쓰는 **MCP 활용 데모**를 완성한다.

- 관객/이해관계자에게 “LLM이 지어낸 값이 아니라 **MCP JSON-RPC `tools/call`** → 엔진 결정론 계산”임을 화면·Network 탭에서 증명한다.
- 기존 stdio MCP(Claude Code/Desktop/Inspector)는 유지하고, 브라우저가 붙을 수 있는 **Streamable HTTP 운영 경로**를 추가한다.
- 데모 안정성을 위해 MCP 실패 시 **로컬 엔진 폴백**을 제공한다 (배지로 출처를 정직하게 표시).

## 개발 범위

  - **today = Asia/Seoul 달력일(D14)** 로 `calendar_day_info` 호출
  - 로딩/에러/출처 배지 · 오락용 안전 고지 · (선택) `VITE_MCP_STRICT` 폴백 금지

### MCP 응답 파싱 스펙 (고정 · D16)

성공 시 **`structuredContent`만** 사용한다. `content[0].text` 요약 문자열을 파싱해 간지를 추출하지 않는다.

| 툴 | 간지 경로 | meta 경로 | 정규화 |
|---|---|---|---|
| `saju_palja` | `structuredContent.palja.dayGan`, `structuredContent.palja.dayJi` | `structuredContent.meta.engineVersion`, `ruleVersion` | `ganji = dayGan + dayJi` |
| `calendar_day_info` | `structuredContent.day.dayGanJi` (합쳐진 2글자, 예: `甲申`) | 동일 `structuredContent.meta` | `gan = dayGanJi[0]`, `ji = dayGanJi[1]` (한자 1+1) |

오류 시 (기존 manseryeok-mcp 계약):

| 필드 | 규칙 |
|---|---|
| `isError` | `true` |
| `structuredContent` | **없음** (성공 전용) |
| `content[0].text` | 오류 JSON 문자열 (`code`: `MANSERYEOK_*` / `INVALID_INPUT` / `INTERNAL_ERROR` 등) |
| Provider 동작 | throw → 상위 폴백 또는 에러 UI (`VITE_MCP_STRICT`면 폴백 없이 에러) |

## MCP 전문가 리뷰 후속 보완 항목 (구현 전 반영)

- **R1. Streamable HTTP JSON response 옵션 고정:** `StreamableHTTPServerTransport({ sessionIdGenerator: undefined, enableJsonResponse: true })`를 코드·테스트 기준으로 고정한다. `initialize`/`tools/call` 응답은 `Content-Type: application/json`이어야 하며 `text/event-stream`으로 퇴행하면 실패로 본다.
- **R2. initialize race 방지:** `McpClientProvider`는 `initializePromise` 단일 flight를 공유하고 모든 `tools/call`은 `ensureInitialized()`를 await한다. player/today 병렬 호출 시 initialize 중복·선행 tools/call이 없어야 한다.
- **R3. 오류 파서 fallback:** `isError`의 `content[0].text`는 JSON parse를 먼저 시도하되, SDK 레벨 검증처럼 plain text(`MCP error -32602...`)가 올 때도 crash 없이 raw message/code로 보존한다.
- **R4. 엔진 직접 import 제거 게이트:** setup 경로의 `ManseryeokEngine` 직접 호출은 금지한다. 구현 후 `rg "ManseryeokEngine|getSolarContext" game/src/main.ts` 결과가 비어야 하며, 직접 엔진 접근은 `LocalEngineProvider`에만 둔다.
- **R5. 로컬 엔진 번들 전략:** MCP 데모 기본 경로가 full SDK와 local engine을 정적으로 끌어오지 않도록 `LocalEngineProvider`는 local/fallback 시 동적 import 경로로 둔다. 불가 시 README에 번들 크기·폴백 목적을 명시한다.

## 문서 기반 제약 사항

- 시간대·**오늘 날짜**는 **KST(Asia/Seoul)** 달력일 (D14). MCP/엔진 공통.
- 출생시 미상: `hour: null`, `minute: null`.
- 계산 목적: **오락·자기성찰**. 의료·투자·법률·재무 판단 문구 금지.
- 게임은 간지·오행·meta만 사용. text summary로 간지 추출 금지 (D16).
- stdio stdout 청결 규칙 유지; HTTP는 **별도 프로세스**.
- `createServer()`·툴 등록 로직 재사용; 범위는 transport·클라이언트·데모 UX.
- 팔자 지원 범위 1908-04-01 ~ 2101-12-31. 범위 밖은 에러 UI.
- Local vs MCP 교차 검증: **동일 day ganji** 수준으로 기대 (API 표면 차이·야자시 경계 완전 동일은 계약하지 않음). 데모 고정 입력은 정오 전후(예: 14:30)로 안전하게.
- 서버 로그에 생년월일 전체 dump 금지 (stderr 최소).

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 요구사항이 불명확하면 가능한 케이스를 나누고 확인 후 진행한다.
- 한 기능은 유지보수 가능한 최소 책임 단위로 나누고, 각 단위의 역할과 경계를 계획에 적는다.
- 계획 밖 옵션/확장/부가 기능을 만들기 위한 분리는 하지 않는다.
- 공통화는 계획 범위 안의 중복을 줄일 때만 수행하고 과도한 추상화는 피한다.
- 기존 프로젝트 API, 공식 SDK, 표준 라이브러리를 우선 검토하고 새 의존성은 범위상 필요할 때만 포함한다.
- 데모 성공 정의:
  1. Network(또는 동등 증거)에서 **MCP `initialize` 및 `tools/call`** 이 확인되고
  2. UI 배지에 tool name + engine/rule meta가 보이며
  3. 배틀 완주 가능하고
  4. 서버 다운 시(STRICT 아님) 폴백으로 데모가 죽지 않는다.
