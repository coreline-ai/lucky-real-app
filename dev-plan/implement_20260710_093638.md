## 개발 목적

최근 검토에서 확인된 `web-lucky` 브라우저 스모크 불일치, `mcp-server` HTTP 외부 바인드 인증 리스크, `web-lucky` devDependency audit 경고, `game`/`web-lucky` 번들 크기 경고를 우선순위에 따라 정리하고 구현 가능한 항목을 안전하게 반영한다.

## 개발 범위

- `web-lucky/scripts/browser-check.mjs`가 현재 5모드 허브 구조에서 토정 모드 딥링크를 검증하도록 수정한다.

- MCP HTTP 테스트에 외부 bind 인증 요구 회귀 테스트를 추가한다.
- `web-lucky` 초기 번들 크기를 줄이기 위해 모드 화면을 동적 import로 분리한다.
- `web-lucky` devDependency audit 경고는 production dependency 영향이 없음을 문서화하고, 강제 major upgrade는 이번 범위에서 제외한다.

## 최종 결과 요약

- `web-lucky/scripts/browser-check.mjs`는 `/?mode=tojeong` 딥링크 기준으로 수정되어 현재 허브 구조와 충돌하지 않는다.
- `browser-hub-check.mjs`와 `browser-check.mjs` 모두 안정적인 임시 스크래치 폴더를 사용한다.
- `mcp-server`는 non-local host에서 auth token이 없으면 서버 생성/시작이 실패한다.
- `mcp-server` 회귀 테스트는 108개 통과했고, 외부 bind 무토큰 실패 케이스가 추가됐다.
- `web-lucky/src/main.ts`는 모드별 동적 import 구조로 바뀌어 초기 라우터 chunk가 약 4.43 kB 수준으로 분리됐다.
- `web-lucky` production audit와 `mcp-server` production audit는 모두 0 vulnerabilities 상태다.
- 최종 검증: `git diff --check`, `npm --prefix mcp-server test`, `npm --prefix web-lucky test`, `npm --prefix web-lucky run build`, `npm --prefix game run build` 통과.

## 잔여 리스크 / 후속 과제

- `web-lucky` 전체 `npm audit`의 devDependency 경고는 Vite major upgrade 검증이 필요하다.
- `web-lucky` 엔진 chunk 자체는 여전히 클 수 있으므로 공개 웹 전환 시 worker 분리 또는 엔진 API 경량화가 필요하다.
- Internet-facing MCP 배포는 reverse proxy/gateway rate limit, TLS, 로그/알림, secret manager 구성이 별도로 필요하다.
