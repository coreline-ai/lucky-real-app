# implement_20260713_103828.md

작성 일시: `2026-07-13 10:38:28 KST`

이 문서는 `web-lucky`와 `web-mcp-daily`의 목적을 분리한 상태로 각각 고도화하기 위한 개발 계획이다. 두 앱은 같은 만세력 엔진 생태계를 쓰지만 제품 목적이 다르다. 따라서 통합하지 않고, `web-lucky`는 브라우저 엔진 직접 실행 데모로, `web-mcp-daily`는 MCP 서버 실전 연동 데모로 유지한다.

## 개발 목적

### Track A — web-mcp-daily: MCP 실전 연동 데모

`web-mcp-daily`의 목적은 `manseryeok-mcp` 서버가 실제 외부 클라이언트, PlayMCP 콘솔, 브라우저 앱에서 안전하게 호출될 수 있음을 검증하는 것이다. 브라우저는 엔진을 직접 import하지 않고 `/mcp` JSON-RPC `tools/call`만 사용해야 한다.

핵심 목표:

- PlayMCP 등록 전 사전 검증 흐름을 앱/문서/스크립트로 고정한다.
- 인증, CORS, response size, error classification, tool trace를 명확히 표시한다.
- `tools/list` 크기와 tool description 품질을 관리한다.
- MCP 서버 OFF 상태에서는 fallback 없이 실패해야 한다.
- 공개 HTTPS 임시 검증과 운영 배포 준비를 분리한다.

### Track B — web-lucky: 엔진 직접 실행 웹 데모

`web-lucky`의 목적은 `manseryeok-engine`이 브라우저에서 서버 없이 직접 계산될 수 있음을 보여주는 것이다. MCP 서버 호출 방식으로 바꾸지 않고, 오프라인/PWA 가능성을 살리는 방향으로 최적화한다.

핵심 목표:

- 엔진 root import를 줄이고 기능별 import 경계를 만든다.
- `lunar-solar.generated.json`, `solar-terms.generated.json` 같은 대형 데이터를 필요한 범위만 로드하도록 준비한다.
- 공개 웹 배포 전 큰 chunk 경고를 실질적으로 줄인다.
- 계산이 무거운 기능은 Web Worker 이전 가능성을 검증한다.
- 서버 없이 동일 결과를 재현하는 엔진 쇼케이스 성격을 유지한다.

### Track C — 교차검증

두 앱을 통합하지 않는다. 대신 같은 입력에 대해 `web-lucky`의 엔진 직접 계산 결과와 `web-mcp-daily` 또는 `mcp-server`의 MCP tool 결과가 일치하는지 검증한다.

핵심 목표:

- 같은 fixture 입력으로 엔진 직접 경로와 MCP 경로의 결과 일치를 확인한다.
- 불일치 발생 시 앱 UI 문제가 아니라 엔진 wrapper, schema mapping, 날짜 정책 차이 중 어디인지 분류한다.
- 이 교차검증을 MCP 신뢰도 증거로 사용한다.

## 공통 진행 규칙

- 두 앱의 목적을 섞지 않는다.
- 구현 단위는 유지 가능한 최소 책임 단위로 나눈다.
- 각 Phase는 자체 테스트가 통과한 뒤 다음 Phase로 이동한다.
- 성능 개선은 실제 build 산출물 수치로 판단한다.
- `web-mcp-daily`에서 `manseryeok-engine`, `../engine`, `engine/src` direct import를 금지한다.
- `web-lucky`는 MCP 서버가 없어도 계산 가능해야 한다.
- auth token은 저장소, localStorage, sessionStorage, 로그에 남기지 않는다.
- 큰 응답이나 큰 번들은 경고를 숨기지 말고 budget과 원인으로 관리한다.

## 현재 관찰 결과

- `web-lucky`는 화면별 dynamic import가 이미 적용되어 있다.

  - `lunar-solar-*.js`: 약 3.8MB
  - `index-*.js`: 약 760KB
- `web-mcp-daily`는 MCP-only 구조와 5개 탭 browser smoke가 이미 동작한다.
- `mcp-server report:sizes` 기준 개별 `tools/call`은 OK, `tools/list`는 약 53KB WATCH다.

## 우선순위 결정

1. **1순위: Track A web-mcp-daily / mcp-server PlayMCP 검증 강화**
   - 현재 프로젝트의 핵심 목표가 MCP 서버 실전 사용과 PlayMCP 등록 가능성 검증이기 때문이다.
   - 공개 웹에서 가벼운 번들과 서버 기반 계산이 더 적합하다.

2. **2순위: Track C 교차검증**
   - MCP 서버가 엔진을 정확히 감싸고 있다는 신뢰 증거가 된다.
   - PlayMCP 등록 전 회귀 방지에도 직접 도움이 된다.

3. **3순위: Track B web-lucky 경량화**
   - 엔진 직접 실행 쇼케이스로 중요하지만, 공개 제품의 주력 경로가 MCP라면 후순위다.
   - 단, 공개 웹으로 확장할 계획이 생기면 data sharding이 최우선이다.
