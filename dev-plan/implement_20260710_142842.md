# implement_20260710_142842.md

작성 일시: `2026-07-10 14:28:42 KST`

이 문서는 현재 `mcp-server`와 MCP 전용 데모 웹앱 `web-mcp-daily`의 상태를 고정하고, 현재 로컬 환경에서 실제로 실행 및 개발 가능한 후속 항목을 정리하기 위한 개발 계획이다.

## 개발 목적

현재 프로젝트는 MCP 서버 자체 구현, PlayMCP 등록 전 검증 대시보드, `game` MCP 데모, 그리고 신규 `web-mcp-daily` 웹앱까지 로컬 환경에서 검증 가능한 상태에 도달했다. 이제 필요한 것은 기능을 더 늘리는 것이 아니라, 현재 로컬에서 확인 가능한 MCP 실사용 흐름을 안정화하고, 외부 배포/PlayMCP 등록 전까지 할 수 있는 작업과 할 수 없는 작업을 명확히 나누는 것이다.

핵심 목표는 다음이다.

- 현재 MCP 서버 실행 상태와 데모 웹앱 상태를 문서화한다.
- 로컬에서 사용자가 직접 확인할 수 있는 테스트 절차를 고정한다.
- `web-mcp-daily`가 MCP 서버를 실제로 쓰고 있음을 화면과 테스트에서 더 명확히 만든다.
- 현재 환경에서 개발 가능한 항목과 외부 조건이 필요한 항목을 분리한다.
- 후속 작업이 목적 밖으로 확장되지 않도록 개발 범위를 제한한다.

## 현재 MCP 상태

현재 확인된 로컬 상태는 다음과 같다.

- MCP 서버는 `127.0.0.1:3100`에서 실행 중이다.
- MCP endpoint는 `POST http://127.0.0.1:3100/mcp`이다.
- `/health` 응답은 `ok: true`이다.
- transport는 `streamable-http`이다.
- mode는 `stateless-json`이다.
- engineVersion은 `0.1.0`이다.
- ruleVersion은 `krlt-yaja-2026.07`이다.
- 현재 실행 서버는 로컬 테스트용으로 `MCP_AUTH_TOKEN` 없이 동작한다.
- 로컬 auth 없음은 개발 환경에서 정상이며, 외부 공개/PlayMCP 등록 환경에서는 token 구성이 필요하다.

현재 MCP 서버가 제공하는 주요 검증 표면은 다음이다.

- `/health`
- `/dashboard`
- `/mcp initialize`
- `/mcp tools/list`
- `/mcp tools/call`
- auth header 호환: `X-MCP-Auth-Token`, `Authorization: Bearer`
- response size report
- production smoke

## 현재 데모 웹앱 상태

`web-mcp-daily/`는 기존 `web-lucky`와 분리된 MCP 전용 데모 웹앱이다.

- 실행 URL: `http://127.0.0.1:5173/`
- Vite proxy: `/mcp` → `http://127.0.0.1:3100/mcp`
- `web-lucky`와 달리 `manseryeok-engine`을 직접 import하지 않는다.
- 브라우저는 MCP JSON-RPC 흐름으로 `initialize`, `notifications/initialized`, `tools/call`을 수행한다.
- MCP 서버가 꺼져 있으면 local engine fallback 없이 오류를 보여준다.
- 화면에는 `로컬 MCP 테스트 모드`와 `테스트 데이터로 바로 실행` 버튼이 있다.
- 테스트 버튼 클릭 시 샘플 출생 정보로 오늘 운세 브리핑을 생성한다.

현재 데모 웹앱이 호출하는 MCP tool은 다음이다.

- `calendar_day_info`
- `saju_palja`
- `saju_full_reading`

현재 화면에서 확인 가능한 항목은 다음이다.

- 오늘 모드
- 오늘의 기운
- 나에게 들어오는 흐름
- 오늘의 행동 제안
- 주의 키워드
- MCP endpoint
- auth mode
- 호출 tool 목록
- HTTP status
- 응답 byte size
- `structuredContent` 파싱 여부
- 오류 분류

## 현재 로컬 실행 방법

MCP 서버는 이미 떠 있으면 그대로 사용한다. 새로 띄울 때는 다음 명령을 사용한다.

```bash
npm --prefix mcp-server run dev:http
```

데모 웹앱은 MCP 서버 기본 CORS 허용 포트인 `5173`에서 실행한다.

```bash
npm --prefix web-mcp-daily run dev -- --host 127.0.0.1 --port 5173
```

브라우저에서 다음 URL을 연다.

```text
http://127.0.0.1:5173/
```

사용자 확인 절차:

1. `테스트 데이터로 바로 실행` 클릭.
2. `MCP 서버를 통해 생성됨` 표시 확인.
3. `MCP 검증 정보`에서 `calendar_day_info`, `saju_palja`, `saju_full_reading` 확인.
4. 각 tool의 HTTP `200`, 응답 크기, `structuredContent` 여부 확인.

자동 검증:

```bash
npm --prefix web-mcp-daily run browser:check
```

이 명령은 다음을 확인한다.

- MCP 서버 ON 상태에서 브리핑 생성 성공.
- MCP 서버 OFF 상태에서 오류 UI 표시.
- 잘못된 auth token은 auth 오류로 분류.
- 정상 auth token은 브리핑 생성 성공.
- MCP tool 3개가 실제 호출됨.

## 현재 환경에서 가능한 항목

현재 바로 진행 가능한 항목:

- `web-mcp-daily` UI 문구 개선.
- `dev:local` 실행 스크립트 추가.
- 수동 auth 테스트 패널 추가.
- CORS troubleshooting 문서화.
- compact/default include 정책 비교.
- `browser:check` 유지보수.
- git 변경 범위 분리 및 커밋 준비.

현재 환경만으로 완료 불가능하거나 외부 조건이 필요한 항목:

- 공개 HTTPS 배포.
- PlayMCP 콘솔 실제 등록.
- 운영 token secret manager 구성.
- 운영 도메인 CORS 확정.
- 외부 사용자를 대상으로 한 안정성 보장.
