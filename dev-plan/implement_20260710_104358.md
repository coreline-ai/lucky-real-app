## 개발 목적

`mcp-server`를 카카오 PlayMCP 등록 준비 상태로 끌어올린다. 목표는 **현재 로컬 개발 환경에서 개발·검증 가능한 영역**을 끝까지 마무리하고, PlayMCP 콘솔 로그인·정식 공개 HTTPS 운영처럼 외부 권한이 필요한 일은 수동 검증 단계로 분리하는 것이다.

최종 목표는 다음 5개 항목의 등록 준비도를 높이는 것이다.

- 공개 HTTPS 배포 전 임시 HTTPS 검증 절차 확보
- PlayMCP 콘솔의 "정보 불러오기" 테스트를 위한 endpoint/auth/runbook 준비
- PlayMCP Key/Token 커스텀 헤더 호환성 확보
- tool description 한영 정리로 LLM tool selection 품질 개선
- 응답 크기 큰 tool의 compact/default 정책 검토와 측정 근거 확보

## 현재 환경에서 가능한 수준

- 가능: 로컬 `mcp-server` build/test/smoke 실행.
- 가능: `cloudflared` 임시 HTTPS 터널을 이용한 외부 URL 생성.
- 가능: `docker` 기반 배포 산출물 또는 운영 runbook 준비.
- 가능: 인증 헤더 호환성, tool description, 응답 크기 측정 스크립트, smoke script 개발.
- 제한: PlayMCP 콘솔 로그인, "정보 불러오기" 버튼 클릭, 임시 등록/심사 요청은 사용자 계정과 브라우저 조작이 필요하므로 수동 검증으로 남긴다.
- 제한: 정식 도메인/DNS/TLS/운영 서버 배포는 외부 인프라 결정이 필요하므로 개발 범위에서 제외한다.

## 개발 범위

- `mcp-server`의 `/mcp` 인증을 PlayMCP Key/Token 등록에 맞게 보강한다.
  - 기존 `Authorization: Bearer <token>` 유지.
  - `X-MCP-Auth-Token: <token>` 추가 허용.
  - 누락/오류 토큰 401 정책 유지.
- PlayMCP 임시 HTTPS 테스트 runbook을 문서화한다.
  - 로컬 서버 실행.
  - `cloudflared tunnel --url http://127.0.0.1:3100`.
  - PlayMCP endpoint/auth 입력값.
  - "정보 불러오기" 성공/실패 확인 기준.
- 등록용 tool description을 한영 병기 또는 영어 우선 구조로 정리한다.
  - tool name, input schema, output schema는 깨지지 않게 유지.
  - 의료/투자/법률/재무 판단 금지 고지를 유지.
- HTTP 기반 tool 응답 크기 측정 스크립트를 추가한다.
  - `tools/list`와 smoke fixture 기반 `tools/call` 결과 byte size를 산출.
  - threshold별 경고를 출력.
  - 큰 응답 후보를 보고서로 남긴다.
- 응답 크기 큰 tool의 compact/default 정책을 검토한다.
  - 먼저 측정 결과를 기준으로 위험 tool을 특정한다.
  - schema-breaking 변경 없이 기본 fixture, README, tool 설명, include/compact 사용 가이드를 우선 보강한다.
  - 필요할 경우에만 기존 옵션을 활용해 기본 호출 예시를 compact하게 조정한다.
- production smoke가 PlayMCP 준비 검증에 쓰일 수 있도록 문서와 명령을 정리한다.

## 제외 범위

- PlayMCP 계정 로그인, 콘솔 직접 조작, 임시 등록/심사 요청 자동화.
- 정식 공개 도메인 구매, DNS 설정, TLS 인증서 발급, 클라우드 서버 생성.
- PlayMCP 심사 통과 보장.
- `engine/` 계산 규칙 변경.
- `realapp`, `game`, `web-lucky` 기능 변경.
- OAuth 구현. 이번 범위는 Key/Token 호환성까지만 다룬다.
- 대형 응답 해결을 위한 엔진 내부 구조 변경, worker 분리, 데이터 모델 대개편.

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.

- 계획 밖 옵션/확장/부가 기능을 만들기 위한 분리는 하지 않는다.
- 공통화는 계획 범위 안의 중복을 줄일 때만 수행하고 과도한 추상화는 피한다.

- stdout은 MCP JSON-RPC 채널과 충돌할 수 있으므로 서버 로그는 stderr만 사용한다.
- token, birth payload 원문, 인증 헤더 값은 로그에 남기지 않는다.
- PlayMCP 등록용 작업은 `mcp-server` 중심으로 제한하고, 무관 앱 파일을 수정하지 않는다.

## 최종 산출물 목표

- `mcp-server`는 PlayMCP Key/Token 커스텀 헤더와 Bearer token 모두 지원한다.
- 20개 tool description은 등록/LLM 선택에 적합한 한영 설명으로 정리된다.
- 응답 크기 보고서로 큰 tool 후보와 compact/default 정책이 명확해진다.
- `cloudflared` 임시 HTTPS로 PlayMCP "정보 불러오기"를 테스트할 수 있는 runbook이 존재한다.
- 정식 공개 배포와 PlayMCP 심사 제출은 외부 인프라/계정 작업으로 명확히 분리된다.

## 잔여 리스크 / 후속 과제

- PlayMCP 콘솔 UI가 변경될 경우 runbook 입력 항목이 달라질 수 있다.
- PlayMCP가 특정 auth header 이름을 제한하면 추가 호환 작업이 필요할 수 있다.
- 정식 배포 시 Cloudflare Tunnel, VPS, container platform 중 하나를 확정해야 한다.
- Internet-facing 운영은 reverse proxy/gateway rate limit, 로그 수집, 알림, secret rotation이 추가로 필요하다.
- 큰 응답 tool의 진짜 사용자 질의 패턴은 smoke fixture보다 클 수 있으므로, 등록 후 실제 실패 로그 기반 추가 최적화가 필요하다.
