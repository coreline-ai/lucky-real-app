# 개발 로드맵 / 백로그

작성 기준: Flutter 스마트폰 앱 + 기존 TypeScript 만세력 엔진 활용  
원칙: 수익화 없음, 엔진 정확도 우선, MVP는 매일 쓰는 경험에 집중

## MVP 범위

- [ ] 생년월일시, 성별, 음력/양력, 윤달, 시간대 입력
- [ ] 기존 만세력 엔진 기반 팔자/오행/십신/세운/월운 계산 연동
- [ ] 오늘 날짜 기준 수호신 산출
- [ ] 홈 오늘 운세 요약 3종 제공: 총운, 관계운, 행동 조언
- [ ] 운세 탭 상세 구성 제공: 총운, 감정, 관계, 일/학업, 컨디션, 주의점, 오행 키워드 (01/02 문서 기준)
- [ ] 부족/과다 오행 기반 루틴 추천
- [ ] 일일 접속 또는 루틴 완료 기반 무료 카드 획득
- [ ] 카드 도감 화면
- [ ] 두 사용자 프로필 기반 궁합/케미 요약
- [ ] 공유 이미지 또는 공유 텍스트 생성
- [ ] 로컬 저장: 프로필, 카드 획득 상태, 루틴 완료 상태
- [ ] 개인정보 최소화와 공유 전 민감정보 제거
- [ ] 오락/자기성찰용 안내 문구

## 제외 범위

- [ ] 광고
- [ ] 구독
- [ ] 인앱결제
- [ ] 유료 카드팩
- [ ] 프리미엄 운세
- [ ] 유료 궁합 리포트
- [ ] 유료 테마
- [ ] 전문가 상담 연결
- [ ] 결제/영수증/스토어 상품 관리
- [ ] 랭킹, 경쟁형 PvP, 소셜 피드
- [ ] 의료/투자/법률 조언처럼 오해될 수 있는 해석
- [ ] 전문 감정 판매 수준의 정확도 보증 문구

## 단계별 개발 순서

### Phase 1. 제품 범위와 엔진 계약 고정

- [ ] MVP 사용자 흐름 정의: 프로필 생성 -> 오늘 화면 -> 루틴 -> 카드 -> 공유
- [ ] Flutter 앱과 만세력 엔진 사이 JSON 계약 정의
- [ ] 엔진 응답 모델 정의: 팔자, 오행 분포, 오늘 운, 수호신 후보, 궁합 입력
- [ ] 오류 모델 정의: 지원 범위 밖 날짜, 존재하지 않는 민간시각, 중복 민간시각, 필수 입력 누락
- [ ] 앱에서는 명리 계산 로직을 재구현하지 않고 EngineGateway만 사용하도록 경계 설정

자체 테스트 (npm 명령은 `engine/` 디렉터리에서 실행):

- [ ] 대표 생년월일 5건으로 엔진 응답 스냅샷 생성
- [ ] 음력/윤달/출생시 미상/야자시 경계 케이스 포함
- [ ] `npm run type-check`
- [ ] `npm run build`
- [ ] `npm test`
- [ ] `npm run test:engine`
- [ ] `npm run solar-terms:verify`

### Phase 2. Flutter 앱 골격

- [ ] Flutter 프로젝트 구조 구성
- [ ] 라우팅 구성: 온보딩, 프로필 입력, 오늘, 루틴, 카드 도감, 궁합, 설정
- [ ] 상태관리 선택 및 적용
- [ ] 로컬 저장소 구성
- [ ] 공통 디자인 토큰 구성: 오행 색상, 카드 등급, 수호신 아이콘 체계
- [ ] EngineGateway 인터페이스와 MockGateway 구현

자체 테스트:

- [ ] `flutter pub get`
- [ ] `dart format --set-exit-if-changed .`
- [ ] `flutter analyze`
- [ ] `flutter test`
- [ ] 작은 화면/큰 글자 설정에서 주요 버튼 텍스트가 깨지지 않음

### Phase 3. 프로필 입력과 만세력 결과 연동

- [ ] 생년월일시 입력 UI
- [ ] 양력/음력/윤달 선택 UI
- [ ] 성별 또는 전통 계산용 선택값 UI
- [ ] 출생시 모름 처리
- [ ] 시간대 Asia/Seoul 고정 처리 (선택 구조는 v2)
- [ ] 엔진 오류 메시지 사용자 친화 처리
- [ ] 계산 결과 캐싱

수용 기준:

- [ ] 사용자는 1분 안에 첫 프로필을 만들 수 있다.
- [ ] 필수 입력 누락 시 저장되지 않는다.
- [ ] 지원 범위 밖 날짜는 명확한 안내를 표시한다.
- [ ] 엔진 오류 원문이나 스택트레이스가 노출되지 않는다.
- [ ] 프로필 삭제가 가능하다.

### Phase 4. 오늘의 수호신/오늘 운세

- [ ] 오늘 날짜 기준 세운/월운/일진 조합 계산
- [ ] 오행 균형과 오늘 운 조합으로 수호신 산출
- [ ] 오늘 운세 카드 구성: 총운, 관계운, 주의점, 추천 행동
- [ ] 매일 자정 이후 콘텐츠 갱신
- [ ] 같은 날 같은 프로필은 동일 결과 유지

수용 기준:

- [ ] 앱 재실행 후에도 오늘 결과가 유지된다.
- [ ] 날짜가 바뀌면 오늘 콘텐츠가 갱신된다. 앱 사용 중 자정 통과도 다음 조회에서 갱신된다.
- [ ] 기기 시간을 되돌려도 일일 보상이 중복 지급되지 않는다.
- [ ] 수호신 산출 이유가 한두 문장으로 설명된다.
- [ ] 과장된 예언, 공포 유도, 결제 유도 문구가 없다.

### Phase 5. 오행 루틴

- [ ] 목/화/토/금/수 루틴 템플릿 작성
- [ ] 사용자 오행 상태에 따른 추천 루틴 매칭
- [ ] 오늘 루틴 완료 체크
- [ ] 루틴 완료 기록 저장
- [ ] 7일 흐름 요약

수용 기준:

- [ ] 루틴은 5분 이내 실천 가능한 행동으로 제안된다.
- [ ] 완료 상태는 앱 재실행 후에도 유지된다.
- [ ] 루틴은 건강/의료 효과를 보증하지 않는다.
- [ ] 오늘 운세와 루틴의 메시지가 서로 모순되지 않는다.

### Phase 6. 카드/도감 보상

- [ ] 수호신 카드 모델 정의
- [ ] 카드 획득 조건 정의: 첫 프로필 생성, 일일 확인, 루틴 완료, 궁합 공유
- [ ] 중복 카드 처리 정책 정의
- [ ] 카드 상세 화면
- [ ] 도감 진행률 표시
- [ ] 모든 보상은 무료 행동 기반으로 제한

수용 기준:

- [ ] 결제 없이 모든 카드가 획득 가능하다.
- [ ] 카드 획득 조건이 사용자에게 명확하다.
- [ ] 중복 획득 시 보상 처리가 일관적이다.
- [ ] 도감 진행률이 실제 보유 카드와 일치한다.

### Phase 7. 궁합/케미 공유

- [ ] 두 번째 프로필 입력 또는 저장된 프로필 선택
- [ ] 오행 상생/상극 기반 케미 요약
- [ ] 관계 타입 선택: 친구, 연인, 가족, 동료
- [ ] 공유용 문구 생성
- [ ] 공유 이미지 생성
- [ ] 공유 전 생년월일시 숨김 기본값 적용

수용 기준:

- [ ] 공유 결과에 민감한 출생정보가 기본 노출되지 않는다.
- [ ] 케미 결과는 단정적 판단이 아니라 경향/대화 소재로 표현된다.
- [ ] 공유 이미지가 주요 메신저에서 잘리지 않는다.
- [ ] 상대 프로필 삭제가 가능하다.

### Phase 8. QA, 스토어 준비, 베타

- [ ] 온보딩부터 공유까지 E2E 테스트
- [ ] Android/iOS 실기기 확인
- [ ] 앱 권한 최소화 확인
- [ ] 개인정보 처리 안내 문구 확인
- [ ] 접근성 확인: 글자 크기, 대비, 스크린리더 라벨
- [ ] 크래시/빈 화면/네트워크 실패 대응 확인
- [ ] 베타 피드백 항목 정리

수용 기준:

- [ ] 신규 사용자가 막힘 없이 첫 수호신을 확인한다.
- [ ] 앱이 결제/광고/구독 권한이나 SDK를 포함하지 않는다.
- [ ] 엔진 실패 시 안전한 안내 화면으로 복구된다.
- [ ] 주요 플로우에서 개인정보가 불필요하게 공유되지 않는다.

## 에픽/스토리/수용기준

### Epic 1. 프로필과 엔진 연동

- [ ] Story: 사용자는 생년월일시와 음양력 정보를 입력해 프로필을 만든다.
- [ ] Story: 앱은 엔진 결과를 받아 오늘 콘텐츠의 원천 데이터로 사용한다.
- [ ] Story: 엔진 계산 불가 케이스는 친절한 오류로 안내된다.

수용 기준:

- [ ] 명리 계산은 앱 UI 레이어에 중복 구현하지 않는다.
- [ ] 엔진 응답은 typed model로 변환된다.
- [ ] 입력값과 결과 캐시는 사용자가 삭제할 수 있다.

### Epic 2. 오늘 경험

- [ ] Story: 사용자는 앱 첫 화면에서 오늘의 수호신을 본다.
- [ ] Story: 사용자는 오늘 운세와 추천 행동을 확인한다.
- [ ] Story: 날짜 변경 시 오늘 콘텐츠가 새로 계산된다.

수용 기준:

- [ ] 오늘 화면은 3초 이내 표시된다.
- [ ] 오늘의 수호신, 운세, 루틴 진입이 첫 화면에서 가능하다.
- [ ] 같은 입력과 같은 날짜는 동일한 결과를 낸다.

### Epic 3. 오행 루틴

- [ ] Story: 사용자는 오늘의 오행 루틴을 받는다.
- [ ] Story: 사용자는 루틴 완료를 체크한다.
- [ ] Story: 사용자는 최근 완료 흐름을 본다.

수용 기준:

- [ ] 루틴은 무료이며 횟수 제한이 없다.
- [ ] 루틴 완료는 보상과 연결된다.
- [ ] 위험하거나 의학적 효과를 암시하는 루틴은 제외된다.

### Epic 4. 카드/도감

- [ ] Story: 사용자는 앱 활동으로 카드를 얻는다.
- [ ] Story: 사용자는 도감에서 보유/미보유 카드를 확인한다.
- [ ] Story: 사용자는 카드 상세 설명을 본다.

수용 기준:

- [ ] 모든 카드는 무료 획득 가능하다.
- [ ] 카드 획득 확률/조건은 숨기지 않는다.
- [ ] 도감 데이터와 저장 데이터가 불일치하지 않는다.

### Epic 5. 궁합/케미 공유

- [ ] Story: 사용자는 두 프로필의 케미를 확인한다.
- [ ] Story: 사용자는 케미 결과를 공유한다.
- [ ] Story: 사용자는 공유 전 민감정보 표시 여부를 제어한다.

수용 기준:

- [ ] 공유 기본값은 개인정보 비노출이다.
- [ ] 결과 문구는 관계를 단정하거나 불안을 조장하지 않는다.
- [ ] 공유 이미지는 Android/iOS 공유 시트에서 정상 전달된다.

## 병렬 개발 워크스트림

| 워크스트림 | 담당 | 주요 산출물 | 충돌 위험 |
|---|---|---|---|
| A. 엔진/계약 | 계산 정확도, 데이터 계약, fixture | EngineGateway JSON schema, 오류 코드, 스냅샷 | 중 |
| B. Flutter 앱 코어 | 라우팅, 상태관리, 로컬 저장, 공통 UI | 앱 골격, DB, 테마, navigation | 중 |
| C. 콘텐츠/해석 | 수호신/운세/루틴/케미 문구 | 안전한 템플릿, 용어 설명 | 낮음 |
| D. 보상/도감 | 카드 모델, 획득 조건, 중복 처리 | 도감 데이터, 무료 보상 루프 | 낮음 |
| E. QA/릴리스 | 테스트, 실기기, 접근성, 개인정보 | QA checklist, release gate | 낮음 |

병렬 작업 규칙:

- 워크스트림 A의 EngineGateway 계약이 먼저 고정되어야 한다.
- B는 MockGateway로 UI를 먼저 진행할 수 있다.
- C/D는 JSON 마스터 데이터 형태로 독립 개발한다.
- E는 각 Phase 종료 시 게이트를 갱신한다.
- 공유 파일을 수정해야 하면 통합 담당자가 조정한다.

- [ ] 프로필 삭제 가능
- [ ] 로컬 저장 데이터 최소화
- [ ] 로그에 출생정보 출력 금지

## 최종 완료 체크리스트

- [ ] MVP 범위가 오늘 경험 중심으로 유지된다.
- [ ] 수익화 기능이 전부 제외된다.
- [ ] 기존 수익화 후보가 무료 확장 백로그로 재분류된다.
- [ ] 엔진 계약이 Flutter UI와 분리된다.
- [ ] 주요 날짜/음력/법정시 경계 케이스가 검증된다.
- [ ] 카드와 루틴은 무료 사용 행동으로만 보상된다.
- [ ] 공유 기능은 개인정보 보호 기본값을 가진다.
- [ ] QA 게이트 통과 전 다음 Phase로 넘어가지 않는다.
