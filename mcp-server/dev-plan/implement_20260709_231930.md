작성 일시: `2026-07-09 23:19:30 KST`

이 문서는 이번 개발의 범위를 고정하고, 구현이 목적 밖으로 확장되지 않도록 하기 위한 작업 문서다.

## 개발 목적

`mcp-server`의 로컬 **Tool Dashboard** (`GET /dashboard`, `src/dashboard.ts`)를
“동작만 하는 디버그 페이지”에서 **한눈에 통과/실패를 스캔할 수 있는 점검 UI**로 개선한다.

- 기능( tools/list, fixture 기반 tools/call, health )은 유지한다.
- 정보 밀도·시각 계층·상태 피드백·그룹 스캔성을 우선 고친다.
- 대시보드는 **로컬 점검용** 성격을 유지한다 (공개 제품 포털화 금지).

## 개발 범위

### 현재 문제 (고정)

| 이슈 | 현상 |
|---|---|
| 스캔 불가 | 한 행에 description + payload JSON + result JSON 과밀 |
| 상태 약함 | KPI/행에서 PASS·FAIL이 본문과 비슷한 무게 |
| 그룹 부재 | Calendar/Saju 등이 pill 반복만 있고 섹션/필터 없음 |
| 결과 표현 | 성공해도 JSON dump 위주, 사람용 한 줄 요약 없음 |
| 구조 | HTML/CSS/JS가 `dashboard.ts` 문자열 한 덩어리 → 스타일 개선 비용 큼 |

### 목표 UX (수용 기준)

1. 기본 화면에서 **툴 이름 + 상태(badge) + 소요 ms + 한 줄 요약**이 먼저 보인다.
2. description / payload / 상세 JSON은 **기본 접힘** (펼치기 가능).
3. Failed > 0 이면 Failed KPI가 **시각적으로 강조**된다.
4. 그룹 필터(또는 그룹 섹션)로 20툴을 나눌 수 있다.
5. Run All 시 **진행률(n/20)** 이 보인다.
6. 기존 smoke 성공 기준 유지: HTTP 200 + JSON-RPC result + `isError !== true` + `structuredContent` 존재.
7. `/dashboard` 로컬 기본 활성·외부 bind 시 비활성 정책 유지 (`http.ts` 계약).
8. `npm test` (특히 `tests/http.test.ts` dashboard 관련) 그린.

### 구현 범위

#### Phase A — 응급 미용 + 정보 접기 (필수)

| 항목 | 내용 |
|---|---|
| 파일 분리(권장) | `src/dashboard/` 로 HTML 템플릿·CSS·클라이언트 JS 분리 후 `dashboard.ts`는 fixture export + 템플릿 조립만 (또는 점진 분리) |
| CSS | 행 좌측 상태 바, KPI 색(pass 민트 / fail 로즈 강조), hover, 타이포 weight 정리 |
| 테이블/카드 공통 | description `line-clamp: 2` 또는 기본 숨김+토글 |
| payload | `<details>` **기본 닫힘** 유지·스타일 정돈 |
| result | 성공 시 JSON pre **기본 닫힘**, 한 줄 요약 우선 표시 |
| 요약 로직 | `summarizeResult`를 사람용 한 줄로 개선: 예) `OK · 38ms · {text 앞 N자}` 또는 structured 핵심 키 1개 |

| 진행 | Run All 중 `runningCount/total` 텍스트 또는 progress bar |
| 카피 | 한글 부제: `툴 상태 점검 (로컬)` |

## 문서 기반 제약 사항

- 대시보드는 **로컬 점검 UX**다. 마케팅 랜딩으로 만들지 않는다.
- fixture 출생정보는 UI에 표시될 수 있으나 **서버에 저장하지 않는다** (기존 고지 유지).
- 성공 판정 로직을 느슨하게 바꾸지 않는다 (structuredContent 필수 유지).
- 외부 bind 시 대시보드 노출 정책을 완화하지 않는다.
- 인라인 문자열 유지 시에도 동작 동일해야 하나, **분리를 권장**한다 (유지보수).

## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 요구사항이 불명확하면 가능한 케이스를 나누고 확인 후 진행한다.
- 한 기능은 유지보수 가능한 최소 책임 단위로 나누고, 각 단위의 역할과 경계를 계획에 적는다.
- 계획 밖 옵션/확장/부가 기능을 만들기 위한 분리는 하지 않는다.
- 공통화는 계획 범위 안의 중복을 줄일 때만 수행하고 과도한 추상화는 피한다.
- 기존 프로젝트 API, 공식 SDK, 표준 라이브러리를 우선 검토하고 새 의존성은 범위상 필요할 때만 포함한다.
- 성공 정의: 스캔 가능한 UI + 기존 HTTP/dashboard 테스트 그린 + 수동으로 전체 툴 테스트 1회 통과 확인.
