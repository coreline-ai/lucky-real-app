## 개발 목적

`mcp-server`의 HTTP MCP 서비스를 **운영 배포 전 로컬에서 검증 가능한 수준**까지 보강한다. 이미 구현된 HTTP hardening(health metrics, Content-Length guard, timeout, graceful shutdown, dashboard gate)을 기준선으로 삼고, 다음 단계에서는 외부 노출 전에 필요한 **인증, rate limit, 배포 smoke, CI 게이트, 운영 문서**를 추가한다.

핵심 목표는 실제 클라우드/도메인 없이도 로컬에서 다음을 증명하는 것이다.

- 인증이 켜진 경우 `/mcp`가 무단 호출을 거부한다.
- 과도한 요청이 rate limit으로 차단된다.
- 운영 설정 조합(`HOST=0.0.0.0`, `ENABLE_DASHBOARD=false`, 제한된 CORS 등)이 자동 smoke로 검증된다.
- CI에서 MCP build/test/audit/smoke를 반복 실행할 수 있다.

## 문서 기반 제약 사항

- 현재 `/mcp`는 stateless Streamable HTTP + `enableJsonResponse: true` 모드다.
- 현재 `/dashboard`는 로컬 bind에서 기본 활성화, 외부 bind에서는 `ENABLE_DASHBOARD=true`일 때만 노출된다.
- 현재 body size 제한은 `Content-Length` 기준이다. chunked body counting은 SDK stream 소비와 충돌하지 않도록 이 계획에서도 신중히 다룬다.
- 모든 MCP tool은 로컬 결정론적 계산이며 네트워크 런타임 호출이 없다.

## 잔여 리스크 / 후속 과제

- 실제 운영 DNS/HTTPS/TLS, 방화벽, LB/CDN, process manager 자동 재시작은 로컬 코드만으로 확정할 수 없다.
- multi-instance rate limit은 in-memory 방식으로는 불충분하므로 reverse proxy 또는 gateway rate limit이 필요하다.
- 운영 로그 수집/알림은 배포 환경(CloudWatch, Grafana, Datadog 등) 결정 후 별도 연결한다.
- 인증 token 저장/회전/권한 분리는 secret manager 정책 확정 후 별도 워크스트림으로 진행한다.
