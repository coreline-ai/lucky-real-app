## 공통 진행 규칙

- 각 Phase는 앞선 Phase의 자체 테스트 완료 후에만 시작한다.
- 구현 중 발생한 이슈는 해당 Phase에서 수정하고 기록한다.
- 체크박스 상태를 실제 진행 상태와 맞게 업데이트한다.
- 문서에 없는 범위 확장은 하지 않는다.
- 요구사항이 불명확하면 가능한 케이스를 나누고 확인 후 진행한다.
- 한 기능은 유지보수 가능한 최소 책임 단위로 나누고, 각 단위의 역할과 경계를 계획에 적는다.
- 계획 밖 옵션/확장/부가 기능을 만들기 위한 분리는 하지 않는다.
- 공통화는 계획 범위 안의 중복을 줄일 때만 수행하고 과도한 추상화는 피한다.
- 기존 프로젝트 API, 공식 SDK, 표준 라이브러리를 우선 검토하고 새 의존성은 범위상 필요할 때만 포함한다.
- **(프로젝트 특화)** 엔진 함수 시그니처가 정본이다. 툴 스키마가 엔진과 어긋나면 스키마를 고친다.
- **(프로젝트 특화)** 성공 응답은 `structuredContent`(전체 JSON) + `content`(한 줄 한국어 요약) 이중 반환하고 `meta`를 포함한다. **structuredContent를 반환하는 툴은 반드시 `outputSchema`를 함께 등록한다** (SDK가 양방향 강제: outputSchema 없는 structuredContent, structuredContent 없는 outputSchema 모두 오류 처리됨 — D10).
- **(프로젝트 특화)** 전 툴 annotations `{ readOnlyHint: true, openWorldHint: false }` — 결정론적 로컬 계산, 네트워크 없음.
- **(프로젝트 특화)** stdio 서버의 stdout은 JSON-RPC 채널이다. 서버 코드 어디에서도 `console.log`/stdout 출력 금지, 진단 로그는 `console.error`(stderr)만 사용한다. (엔진 src에는 console 출력이 없음을 확인함)

## 공용 스키마 사양 (P1~P2에서 구현)

```ts
// schemas/birth.ts — 전 툴 공용. realapp 09 계약의 상위집합(진태양시 노출)
BirthInput = {
  year: int, month: 1..12, day: 1..31,
  hour: 0..23 | null (기본 null = 출생시 미상),
  minute: 0..59 | null (기본 null),
  gender: 'male' | 'female',                    // BirthInputData 필수 필드
  calendarType: 'solar' | 'lunar' (기본 solar), // → isLunar 매핑
  isLeapMonth: bool (기본 false, solar와 동시 지정 시 거부),
  midnightMode: 'yaja' | 'joja' (기본 yaja),
  trueSolarTime: bool (기본 false),
  longitude: 124..132 (선택, trueSolarTime=true일 때만 유효),
  birthPlace: string | null (선택),
}
// schemas/palja.ts
PaljaInput = {
  yearGan, yearJi, monthGan, monthJi, dayGan, dayJi: non-empty string,
  hourGan, hourJi: string (기본 '' — 빈 문자열 = 시주 미상, D11)
}
BirthOrPalja = { birth?: BirthInput, palja?: PaljaInput }  // 정확히 하나 필수 (refine)

// schemas/output.ts — 공용 출력 스키마 조각 (D10)
MetaSchema = { engineVersion: string, ruleVersion: string }
// 툴별 outputSchema = { meta: MetaSchema, ...핵심 필드 타입 고정, 대형 중첩부 passthrough }
```

**오류 응답 계약(`errors.ts`)**: 오류는 `isError: true` + `content`(오류 JSON `{ code, message, details }`를 직렬화한 텍스트)로 반환한다. **structuredContent는 성공 응답 전용** — 오류 payload를 structuredContent에 실으면 성공용 outputSchema와 충돌한다(D10). 매핑: `MANSERYEOK_RANGE_ERROR`·`MANSERYEOK_DATA_ERROR`·`AMBIGUOUS_CIVIL_TIME`·`NONEXISTENT_CIVIL_TIME`·`MANSERYEOK_POLICY_ERROR`는 코드·details를 그대로 전달, zod 검증 실패 → `INVALID_INPUT`.

### 불변식 (이 코드베이스에서 반드시 지킬 것)

1. **stdout 출력 금지** — stdio JSON-RPC 채널. 진단은 `console.error`만.
2. **structuredContent ↔ outputSchema 쌍** (D10). 오류 응답은 `isError + content`(JSON 텍스트)만 — structuredContent 금지.
3. **엔진 무수정.** 엔진의 plain Error는 `src/errors.ts`의 `DATA_LOOKUP_ERROR_PATTERNS`로 정규화(현재 6개 throw 지점 전수 커버) — 새 패턴 발견 시 여기에 추가.
4. `interpretResult`·학파 함수는 **서브패스 import** (`manseryeok-engine/engine/...`, 이슈 8) — 네임스페이스에 없음.
5. 시주 미상 = `hourGan/hourJi === ''` (D11). daejeong/hongyeon은 3주 기준 계산이 정상.
6. SDK 입력 zod 검증 실패는 **in-band isError**로 옴(이슈 5) — 프로토콜 rejection을 기대하지 말 것.
7. 날짜 도메인 범위는 zod로 막지 않는다 — 엔진 오류 코드가 계약(P1 방침).
8. 툴/리소스/프롬프트 변경 시 `src/resources/index.ts`의 `TOOL_CATALOG` 갱신 — 드리프트 테스트가 잡아준다. 계획서 체크박스·이슈 기록도 함께 갱신.

### 작업 환경 함정 노트

- zsh `echo`는 `\n` 이스케이프를 해석한다 — 스모크 출력 검사는 `printf`/파일 캡처로(이슈 2).
- 셸 cwd가 세션 간 리셋될 수 있다 — 명령은 `cd <절대경로> && ...` 형태로.
- npm 12 업그레이드 notice가 `tail`을 오염시킨다 — `grep -E "Tests|Files"`로 결과 추출.
- vitest(vite-node)는 `../src/*.js` 지정자를 `.ts`로 해석한다 — NodeNext 관례 유지.
- `manseryeok-engine`은 `file:../engine` 심링크 — 엔진 소스 수정 시 `npm run build`의 prebuild가 재빌드한다.
